# PRD: 二维码扫码快速操作

## 1. 背景与目标

### 为什么需要这个功能
当前 AMS 系统已实现：
- 资产标签打印（生成含 assetCode 的二维码）
- 定期盘点（扫码确认盘点）

但资产**日常**的领用、归还、借用归还操作仍需在 PC 端页面操作。当员工需要快速领用设备、或资产不在现场需要临时记录时，PC 端操作不够便捷。移动端扫码操作可以显著提升这类高频场景的效率。

### 解决什么问题
- 资产领用/归还必须在 PC 端操作，移动场景不方便
- 资产转移调拨时，调出方和调入方需要现场确认
- 紧急维修时，工程师到达现场后需快速确认资产信息
- 盘点功能已有扫码，但日常操作缺失

### 预期效果
- 员工使用手机扫码（扫描资产二维码），快速触发操作菜单（领用/归还/借用归还/查看详情）
- 无需打开电脑，在资产所在地点直接扫码完成操作
- 操作记录与现有流程一致，审批流程不变

---

## 2. 功能范围

### 包含哪些功能点
1. **扫码触发操作菜单**：手机浏览器扫描资产二维码 → 打开 H5 页面 → 显示可执行操作
2. **扫码领用**：扫描资产二维码 → 一键申请领用（需审批）→ 等待审批通知
3. **扫码归还**：扫描资产二维码 → 一键申请归还（需审批）
4. **扫码借用归还**：扫描资产二维码 → 一键归还（属于借用流程的快速归还）
5. **扫码查看详情**：扫描资产二维码 → 打开资产详情页（无需审批）
6. **操作员身份验证**：H5 页面需要扫码人确认身份（自动读取当前登录信息）

### 不包含哪些（明确边界）
- 不包含离线扫码（无网络场景）
- 不包含 RFID 标签支持（仅限二维码）
- 不包含批量扫码（单次单资产操作）
- 不包含扫码预约功能

## 3. 用户故事

### 角色：员工
- 场景：在设备间发现一台闲置的显示器，想快速领用
- 期望：掏出手机扫描设备上的二维码 → 点击「领用」→ 填写领用原因 → 提交申请 → 收到审批通知

### 角色：资产管理员
- 场景：员工 A 临时外借一台投影仪给客户，需要快速归还
- 期望：客户现场扫描投影仪二维码 → 点击「归还」→ 系统记录归还申请

### 角色：维修工程师
- 场景：收到报修，到达现场后需要快速确认设备信息
- 期望：扫描设备二维码 → 查看资产详情（型号、采购日期、保修状态、维修历史）

## 4. 详细需求

### 4.1 二维码格式

与现有标签打印保持一致：
- 编码格式：`ASSET-{assetCode}`（例：`ASSET-A002`）
- 扫码后解析前缀，还原 `assetCode`

### 4.2 后端

#### API

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | /api/scan/{assetCode} | 扫码查询资产基本信息 + 可执行操作列表 |
| POST | /api/scan/{assetCode}/assign | 扫码快速领用（简化参数，基于当前登录人） |
| POST | /api/scan/{assetCode}/return | 扫码快速归还（简化参数，基于当前登录人） |
| POST | /api/scan/{assetCode}/borrow-return | 扫码借用归还 |

#### ScanResponse

```java
{
    "assetId": 1,
    "assetCode": "A002",
    "assetName": "ThinkPad X1 Carbon",
    "category": "HARDWARE",
    "status": "IN_STOCK",  // IN_STOCK / IN_USE / MAINTENANCE
    "assigneeName": null,  // 若 IN_USE，显示当前领用人
    "location": "3F-301",
    "availableActions": ["ASSIGN", "VIEW_DETAIL"],  // 根据状态和权限计算
    // IN_STOCK: [ASSIGN, VIEW_DETAIL]
    // IN_USE (当前人领用): [RETURN, VIEW_DETAIL]
    // IN_USE (他人领用): [VIEW_DETAIL]
    // BORROWED (当前人借用): [BORROW_RETURN, VIEW_DETAIL]
}
```

#### 权限校验
- 扫码领用：资产状态必须为 `IN_STOCK`，且当前人不是该资产 assignee
- 扫码归还：资产状态必须为 `IN_USE`，且当前人是该资产 assignee
- 扫码借用归还：当前人存在有效的 `BORROWED` 状态的借用记录

### 4.3 前端 H5 页面

#### 路由
- `/#/scan/:assetCode` — 扫码操作页（H5 单页应用）

#### 页面布局
```
┌─────────────────────────┐
│  [资产名称]              │
│  编码：A002              │
│  分类：电子设备          │
│  状态：在库              │
│  位置：3F-301            │
├─────────────────────────┤
│  [领 用]  [归 还]        │
│  [查看详情]              │
├─────────────────────────┤
│  操作说明：              │
│  点击按钮完成相应操作     │
└─────────────────────────┘
```

#### 状态流与按钮显示

| 资产状态 | 可执行操作 | 按钮 |
|---------|-----------|------|
| IN_STOCK | 领用 | 显示「领用」按钮 |
| IN_USE（当前人领用） | 归还 | 显示「归还」按钮 |
| IN_USE（他人领用） | 查看详情 | 仅显示「查看详情」 |
| MAINTENANCE | 查看详情 | 仅显示「查看详情」 |
| BORROWED（当前人借用） | 借用归还 | 显示「归还」按钮 |

#### 交互流程
1. 扫码 → 打开 H5 页面 → 显示资产信息 + 可用操作
2. 点击操作按钮 → 确认对话框 → 调用 API → 显示结果
3. 操作成功后显示成功提示（审批中/已完成），可返回首页

### 4.4 与现有审批流集成
- 扫码领用/归还走现有 `ApprovalWorkflow`，保持一致
- 借用归还走现有 `BorrowService.returnBorrowedAsset()`

---

## 5. 验收标准

1. 扫描资产二维码 → 正确打开 H5 页面，显示资产信息和可用操作
2. 资产在库状态 → 点击「领用」→ 提交成功 → 审批流正常触发
3. 资产在用状态（当前人领用）→ 点击「归还」→ 提交成功 → 审批流正常触发
4. 借用中的资产（当前人借用）→ 点击「归还」→ 走借用归还流程
5. H5 页面在手机浏览器中正常显示，按钮适配移动端
6. `mvn compile` 成功，`npm run build` 成功

---

## 6. 优先级

**P2（有益）** — 扫码操作可显著提升移动场景下的资产操作效率，是对现有定期盘点扫码能力的日常化扩展
