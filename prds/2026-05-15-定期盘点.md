# 定期盘点

## 1. 背景与目标

### 为什么需要这个功能
企业固定资产（电脑、设备、仪器等）随着人员调动、业务变更，容易出现实物与账目不符的情况。传统人工盘点费时费力且容易出错，需要一个系统化的定期盘点功能来确保账实一致。

### 解决什么问题
- 资产实物与系统记录不一致，无法真实反映企业资产状况
- 人工盘点效率低，易遗漏、易重复
- 盘点结果无法追溯，无法分析资产流失原因
- 移动盘点依赖纸质清单，体验差

### 预期效果
- 管理员可按部门或资产分类创建季度/月度盘点计划
- 系统自动生成盘点任务清单，分配给盘点负责人
- 盘点人通过移动端扫码快速确认资产在场
- 盘点结束后自动生成差异报告（盘盈/盘亏/正常）

---

## 2. 功能范围

### 包含哪些功能点
1. **盘点计划管理**：创建/编辑/删除盘点计划，支持按部门或资产分类生成盘点范围
2. **盘点任务生成**：根据盘点计划自动生成盘点任务，分配给盘点人
3. **扫码盘点**：盘点人通过移动端扫描资产二维码，确认资产在场
4. **盘点结果记录**：记录每项资产的盘点结果（正常/盘盈/盘亏/待确认）
5. **盘点差异报告**：汇总盘点结果，展示盘盈/盘亏/正常的数量和明细

### 不包含哪些（明确边界）
- 不包含RFID标签打印/管理
- 不包含与财务系统的对账（仅资产层面的实物核对）
- 不包含自动生成盘点报告并发送给邮件（仅在系统内展示）
- 不包含盘盈盘亏的自动账务处理（需人工处理）

---

## 3. 用户故事

### 角色：管理员（ADMIN/MANAGER）
- 场景：季度末，管理员创建「2026年Q2资产盘点计划」，范围选择「研发部」+ 「所有电子设备分类」
- 期望：系统自动生成研发部所有电子设备对应的盘点任务，分配给研发部负责人
- 后续：查看各盘点人提交的结果，下载差异报告

### 角色：盘点负责人（普通员工）
- 场景：收到盘点任务通知，打开移动端，看到需要盘点的资产清单
- 期望：点击每项资产，扫描其二维码，系统标记为「已盘点」
- 后续：提交盘点结果，系统计算差异

### 角色：系统
- 场景：盘点人提交盘点结果
- 期望：系统对比资产台账，自动计算哪些资产「盘盈」（实物多）/「盘亏」（实物少）/「正常」

## 4. 详细需求

### 4.1 页面/组件需求

#### 盘点计划页面（`/#/inventory-plan`）

**计划列表**
- 表格展示：计划名称、盘点范围（部门/分类）、计划日期、状态（待执行/进行中/已完成）、创建人
- 操作：新建、编辑、删除、开始盘点、查看报告

**新建计划表单**
- 计划名称（文本输入）
- 盘点范围类型：按部门 / 按资产分类（二选一）
  - 按部门：多选部门（部门树形选择）
  - 按分类：多选资产分类（下拉多选）
- 计划盘点日期（日期选择）
- 盘点负责人（多选员工）
- 备注（文本域，可选）

**字段**
```typescript
interface InventoryPlan {
  id: number;
  name: string;                    // 计划名称
  scopeType: 'DEPARTMENT' | 'CATEGORY';
  departmentIds?: number[];         // 按部门时
  categoryIds?: number[];           // 按分类时
  planDate: string;                 // 计划盘点日期
  status: 'PENDING' | 'IN_PROGRESS' | 'COMPLETED';
  creatorId: number;
  createdAt: string;
}
```

#### 盘点任务页面（`/#/inventory-task`）

**任务列表**（按计划分组）
- 表格展示：资产编码、资产名称、所属部门、计划名称、盘点状态（待盘点/已盘点）、盘点人、盘点时间
- 状态筛选：全部 / 待盘点 / 已盘点

**扫码盘点**
- 移动端友好：点击「扫码盘点」按钮，调用浏览器摄像头扫描资产二维码（编码格式：`ASSET-{assetCode}`）
- 扫描成功后：
  - 若该资产在当前用户的盘点任务中 → 标记为「已盘点」，记录盘点时间
  - 若不在 → 提示「该资产不在您的盘点清单中」
- 也支持手动选择资产进行盘点

**盘点状态枚举**
```typescript
type InventoryTaskStatus = 'PENDING' | 'CHECKED';
```

#### 盘点记录表（InventoryRecord）
```typescript
interface InventoryRecord {
  id: number;
  taskId: number;           // 关联盘点任务
  assetId: number;         // 资产ID
  assetCode: string;       // 资产编码
  assetName: string;       // 资产名称
  departmentId: number;
  departmentName: string;
  result: 'NORMAL' | 'SURPLUS' | 'MISSING' | 'PENDING';  // 正常/盘盈/盘亏/待确认
  checkedBy: number;       // 盘点人
  checkedAt: string;       // 盘点时间
  remark?: string;         // 备注（盘盈/盘亏说明）
}
```

#### 盘点差异报告页面（`/#/inventory-report/:planId`）

**报告概览**
- 统计卡片：应盘资产数 / 已盘资产数 / 正常数 / 盘盈数 / 盘亏数
- 进度条：已完成百分比

**差异明细**
- Tab页切换：全部 / 正常 / 盘盈 / 盘亏
- 表格：资产编码、资产名称、所属部门、结果、盘点人、盘点时间、备注

**导出**：支持导出Excel（所有记录）

### 4.2 API/后端需求

#### 实体

**InventoryPlan（盘点计划）**
```java
@Entity
@Table(name = "inventory_plan")
public class InventoryPlan {
    Long id;
    String name;
    String scopeType;           // DEPARTMENT / CATEGORY
    String departmentIds;        // JSON array stored as string: "[1,2,3]"
    String categoryIds;          // JSON array stored as string: "[1,2]"
    LocalDate planDate;
    String status;              // PENDING / IN_PROGRESS / COMPLETED
    Long creatorId;
    LocalDateTime createdAt;
}
```

**InventoryTask（盘点任务）**
```java
@Entity
@Table(name = "inventory_task")
public class InventoryTask {
    Long id;
    Long planId;                // 关联盘点计划
    Long assetId;               // 关联资产
    Long assigneeId;            // 盘点负责人
    String status;               // PENDING / CHECKED
    LocalDateTime checkedAt;
}
```

**InventoryRecord（盘点记录）**
```java
@Entity
@Table(name = "inventory_record")
public class InventoryRecord {
    Long id;
    Long taskId;
    Long assetId;
    String assetCode;
    String assetName;
    Long departmentId;
    String departmentName;
    String result;              // NORMAL / SURPLUS / MISSING / PENDING
    Long checkedBy;
    LocalDateTime checkedAt;
    String remark;
}
```

#### Controller 接口

**InventoryPlanController** (`/api/inventory-plans`)
- `GET /` — 分页列表（支持状态筛选）
- `POST /` — 创建盘点计划，同时生成对应的 InventoryTask
- `PUT /{id}` — 更新计划（状态变更时同步任务状态）
- `DELETE /{id}` — 删除计划（同时删除任务和记录）
- `POST /{id}/start` — 开始盘点（状态 PENDING → IN_PROGRESS）
- `POST /{id}/complete` — 完成盘点，生成差异报告数据

**InventoryTaskController** (`/api/inventory-tasks`)
- `GET /` — 分页列表（支持 planId、status、assigneeId 筛选）
- `GET /my` — 获取当前用户的盘点任务
- `PUT /{id}/check` — 标记资产已盘点（扫码或手动）
- `PUT /{id}/uncheck` — 取消盘点标记（重新盘点）

**InventoryRecordController** (`/api/inventory-records`)
- `GET /` — 分页列表（支持 planId、result 筛选）
- `GET /report/{planId}` — 获取指定计划的盘点报告汇总
- `GET /report/{planId}/export` — 导出Excel

#### Service 逻辑

**InventoryPlanService.createPlan()**
1. 保存 InventoryPlan
2. 根据 scopeType 查询对应资产（部门过滤或分类过滤）
3. 为每个资产创建 InventoryTask，assigneeId = 当前用户（或计划的盘点负责人）
4. 返回创建的任务数量

**InventoryTaskService.checkAsset()**
1. 查找 task（assetId + assigneeId 双重校验）
2. 更新 task.status = CHECKED，checkedAt = now
3. 创建 InventoryRecord（result = NORMAL）
4. 返回更新后的 task

**InventoryPlanService.completePlan()（核心差异计算）**
盘点完成后，扫描所有 task：
- 有 CHECKED 记录 → result = NORMAL
- 无 CHECKED 记录 → result = MISSING（盘亏）
- 额外存在的资产（扫码时发现资产不在原始任务列表）→ result = SURPLUS（盘盈），需新建 task 和 record

## 5. 验收标准

### 功能上线判定条件
1. 管理员能够创建「按部门」的盘点计划，系统自动生成对应部门的资产任务
2. 管理员能够创建「按分类」的盘点计划，系统自动生成对应分类的资产任务
3. 盘点人能够看到自己的盘点任务列表
4. 盘点人能够通过扫码确认资产在场，任务状态从「待盘点」变为「已盘点」
5. 盘点完成后，管理员能够查看差异报告，包含正常/盘盈/盘亏的统计数据
6. 管理员能够导出盘点报告为Excel

### 验证方法
- 创建计划：POST `/api/inventory-plans`，检查返回的 planId 和生成的任务数量
- 扫码盘点：PUT `/api/inventory-tasks/{id}/check`，验证 task 状态和 record 生成
- 差异报告：POST `/api/inventory-plans/{id}/complete`，GET `/api/inventory-records/report/{planId}` 验证统计正确
- 导出：GET `/api/inventory-records/report/{planId}/export` 返回 Excel 文件

---

## 6. 优先级

- **P1（重要）**
- 理由：定期盘点是资产全生命周期管理的基础功能，确保账实一致是企业固定资产管理的核心合规要求。没有盘点功能，资产系统只能记录静态台账，无法反映资产真实变动情况。
