# 维保到期提醒

## 1. 背景与目标

### 为什么需要这个功能
资产管理系统中的固定资产（电脑、打印机、设备等）通常有明确的维保期限（warrantyEnd）。随着时间推移，资产维保陆续到期，若无系统化提醒，管理员往往在资产故障报修时才得知已过保，错失续保或预防性维护的最佳时机。

### 解决什么问题
- 维保到期无预警，过保后产生高额维修费用
- 管理员无法主动掌握哪些资产即将到期
- 资产维保状态不透明，管理层无法据此做采购/续保决策

### 预期效果
- 仪表盘展示"30天内即将过保的资产"卡片，支持筛选天数
- 后台定时任务每日扫描 warrantyEnd 字段，提前 N 天（默认30天）触发系统通知
- 通知推送给资产当前使用人（assignee）或管理员

## 2. 功能范围

### 包含哪些功能点
1. **仪表盘维保卡片**：展示30天内即将到期的资产列表，支持按天数筛选（7/15/30/60天）
2. **定时扫描任务**：每天早上8点（可配置）扫描所有未删除资产的 warrantyEnd 字段
3. **维保到期通知**：发现即将到期资产后，创建 ASSET_EXPIRING_WARRANTY 类型的通知，推送给 assignee 或管理员
4. **去重通知机制**：同一资产在同一批次只发送一次通知（已通知过的资产不重复通知）

### 不包含哪些（明确边界）
- 不包含维保续期/维保合同管理（仅提醒，不做续保流程）
- 不包含邮件/SMS/钉钉等外部通知渠道（仅使用系统内通知）
- 不包含按资产分类筛选维保即将到期的功能

## 3. 用户故事

### 角色：资产使用人（普通员工）
- 场景：名下的某台笔记本电脑还有15天维保到期
- 期望：收到系统通知"资产「笔记本电脑」（编码：A002）的维保将于2026-06-01到期，请及时处理"
- 后续：可联系IT部门续保或安排数据迁移

### 角色：管理员（ADMIN / MANAGER）
- 场景：登录仪表盘，看到"维保即将到期"卡片显示3条记录
- 期望：点击卡片展开列表，看到每条资产的名称、编码、到期日期、分类
- 后续：可导出清单，统一安排续保采购

### 角色：系统
- 场景：每日早上8点，定时任务扫描所有资产
- 期望：对于 warrantyEnd 在未来30天内且未发送过通知的资产，自动创建通知

## 4. 详细需求

### 4.1 页面/组件需求

#### 仪表盘页面（`/#/dashboard`）

**维保即将到期卡片**
- 位置：现有仪表盘统计卡片下方，独立一行
- 内容：
  - 卡片标题："维保即将到期" + 红色徽章显示数量
  - 默认显示30天内到期的资产
  - 支持天数切换：7天 / 15天 / 30天 / 60天
  - 列表展示：资产名称、资产编码、分类、到期日期
  - 每行右侧显示"剩余X天"标签（绿色=<15天，橙色=15-30天，红色=>30天或已到期）
  - 点击"查看全部"跳转至筛选后的资产列表页（待后续实现，目前仅展示）

### 4.2 API/后端需求

#### 现有组件（已实现）

**WarrantyNotificationService** (`com.ams.service.WarrantyNotificationService`)
- `@Scheduled(cron = "0 0 8 * * ?")` 每日早8点执行
- `getExpiringWarrantyAssets(int days)` 查询指定天数内到期的资产
- `sendExpiringWarrantyNotifications(int days)` 发送通知（带去重）
- 使用 `notifiedAssetIds` Set 做内存级去重

**DashboardController** (`com.ams.controller.DashboardController`)
- `GET /api/dashboard/expiring-warranty?days=30` 返回即将到期的资产列表

**ExpiringWarrantyResponse** (`com.ams.dto.ExpiringWarrantyResponse`)
```java
private Long id;
private String assetCode;    // 资产编码
private String name;         // 资产名称
private String category;     // 分类
private LocalDate warrantyEnd; // 维保到期日
```

#### 无需新增后端接口

### 4.3 前端需求

#### 类型定义 (`frontend/src/types/index.ts`)
```typescript
export interface ExpiringWarranty {
  id: number;
  assetCode: string;
  name: string;
  category: string;
  warrantyEnd: string;
}
```

#### API Hook (`frontend/src/api/asset.ts`)
```typescript
export function useExpiringWarranty(days: number = 30) {
  return useQuery({
    queryKey: ['dashboard', 'expiring-warranty', days],
    queryFn: async () => {
      const { data } = await http.get<ExpiringWarranty[]>('/dashboard/expiring-warranty', { params: { days } });
      return data;
    },
  });
}
```

#### 仪表盘组件 (`frontend/src/pages/Dashboard.tsx`)
- 新增 `ExpiringWarrantyCard` 组件（内嵌于 Dashboard）
- 使用 `useExpiringWarranty(days)` 获取数据
- Segmented 分页器选择：7天 / 15天 / 30天 / 60天
- 列表最多显示5条，超出显示"还有X条"
- 颜色规则：剩余天数<=7红色，8-15橙色，16-30黄色，>30绿色

## 5. 验收标准

### 功能上线判定条件
1. 仪表盘"维保即将到期"卡片正确显示30天内到期的资产数量
2. 切换天数筛选（7/15/30/60天），列表数据正确刷新
3. 每行显示资产名称、编码、分类、到期日期、剩余天数标签
4. 定时任务在应用启动后每日8点执行（通过日志验证）
5. 相同资产在同一批次不会重复发送通知（去重机制有效）

### 验证方法
- 后端：启动应用，查看日志中 "Running warranty expiration check..." 和 "Warranty check completed. X notifications sent."
- 前端：访问 `/#/dashboard`，确认卡片显示在仪表盘上
- 切换天数筛选，确认列表数据变化
- 查看 NotificationList 页面，确认有 ASSET_EXPIRING_WARRANTY 类型的通知

## 6. 优先级

- **P1（重要）**
- 理由：维保到期提醒是资产全生命周期管理的关键环节，直接影响企业运维成本和资产使用效率。定时通知可以显著减少"事后救火"式的维修经历。
