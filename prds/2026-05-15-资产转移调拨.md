# PRD：资产转移调拨

## 1. 背景与目标

### 背景
在中小企业日常运营中，资产在部门之间转移是高频需求（例如：人员调动时资产跟着人走、设备从A部门调拨到B部门）。当前 AMS 系统只支持"领用"和"归还"，无法记录资产在部门之间的转移历史，导致：
- 资产归属不清晰
- 审计时无法追溯资产流转路径
- 部门资产统计不准确

### 目标
新增「资产转移调拨」功能，实现：
- 发起资产调拨申请（可批量选择多个资产）
- 记录原部门、新部门、转移原因、操作人
- 审批通过后，自动更新资产的归属部门（assignee）
- 保留完整的资产调拨历史记录

---

## 2. 功能范围

### 核心功能
1. **调拨申请**：员工选择资产，发起调拨申请，填写目标部门/员工、转移原因、计划调拨日期
2. **调拨审批**：类似借用/领用的审批流程，支持审批人通过或驳回
3. **审批通过后**：自动更新 `Asset.assignee`（归属员工）和对应的 `Asset.status`，记录 `AssetTransferRecord`
4. **调拨记录查询**：查看资产完整的调拨历史

### 边界说明
- 调拨的资产状态必须为 `IN_USE`（已领用状态），不能调拨在库或维修中的资产
- 同一时间一个资产只能有一个待处理的调拨申请（防止重复申请）
- 调拨记录永久保留，不可删除
- 支持批量调拨（多个资产同时调拨到同一目标部门/员工）

---

## 3. 用户故事

| 角色 | 场景 | 期望结果 |
|------|------|----------|
| 员工 | 想把手中资产调拨到另一部门 | 发起调拨申请，填写目标部门和原因，等待审批 |
| 部门经理 | 收到调拨申请 | 审批通过/驳回，附上备注 |
| 审批通过后 | 系统自动 | 资产 assignee 更新，状态不变，调拨记录写入 |
| 资产管理员 | 查询某资产的流转历史 | 查看完整的调拨时间线（谁→谁，何时，何原因） |
| 财务/审计 | 资产清查时 | 导出资产调拨记录作为流转凭证 |

## 4. 详细需求

### 4.1 后端

#### 实体设计

**AssetTransferRecord**（资产调拨记录）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 主键 |
| `assetId` | Long | 关联资产ID |
| `assetCode` | String | 资产编码（冗余存储，便于查询） |
| `assetName` | String | 资产名称（冗余存储） |
| `fromEmployeeId` | Long | 原归属员工ID |
| `fromEmployeeName` | String | 原归属员工姓名（冗余） |
| `fromDeptId` | Long | 原部门ID |
| `fromDeptName` | String | 原部门名称（冗余） |
| `toEmployeeId` | Long | 新归属员工ID |
| `toEmployeeName` | String | 新归属员工姓名（冗余） |
| `toDeptId` | Long | 新部门ID |
| `toDeptName` | String | 新部门名称（冗余） |
| `transferDate` | LocalDate | 计划调拨日期 |
| `reason` | String | 调拨原因 |
| `status` | TransferStatus | PENDING / APPROVED / REJECTED |
| `remark` | String | 审批备注（驳回原因） |
| `applicantId` | Long | 申请人ID |
| `applicantName` | String | 申请人姓名 |
| `approverId` | Long | 审批人ID（审批时填写） |
| `approverName` | String | 审批人姓名 |
| `approvalId` | Long | 关联的 ApprovalRequest ID |
| `createdAt` | LocalDateTime | 申请时间 |
| `updatedAt` | LocalDateTime | 更新时间 |

枚举 **TransferStatus**：`PENDING`、`APPROVED`、`REJECTED`

#### 审批类型
新增 `ApprovalType.TRANSFER`

#### API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/transfers/apply` | 发起调拨申请 |
| GET | `/api/transfers` | 查询调拨记录（支持分页、资产编码、状态筛选） |
| GET | `/api/transfers/{id}` | 查询单条调拨记录详情 |
| GET | `/api/transfers/pending` | 查询当前用户待审批的调拨申请 |
| GET | `/api/transfers/history` | 查询当前用户的历史调拨记录（作为申请人） |
| POST | `/api/transfers/{id}/approve` | 审批通过 |
| POST | `/api/transfers/{id}/reject` | 审批驳回（需备注原因） |
| GET | `/api/transfers/asset/{assetId}` | 查询某资产的完整调拨历史 |

#### DTO

**TransferApplyRequest**：
```java
{
  "assetIds": [1, 2, 3],          // 批量资产ID
  "toEmployeeId": 10,             // 目标员工
  "toDeptId": 5,                  // 目标部门
  "transferDate": "2026-05-20",   // 计划调拨日期
  "reason": "人员调动"             // 调拨原因
}
```

**TransferRecordResponse**：
```java
{
  "id": 1,
  "assetCode": "A001",
  "assetName": "ThinkPad X1",
  "fromEmployeeName": "张三",
  "fromDeptName": "技术部",
  "toEmployeeName": "李四",
  "toDeptName": "销售部",
  "transferDate": "2026-05-20",
  "reason": "人员调动",
  "status": "PENDING",
  "remark": null,
  "applicantName": "张三",
  "createdAt": "2026-05-15T10:00:00"
}
```

### 4.2 前端

#### 页面
1. **TransferList.tsx**（调拨记录列表页）
   - Tab：待我审批 / 我发起的 / 全部记录
   - 列表字段：资产编码、资产名称、原归属、新归属、调拨日期、状态、操作
   - 支持按资产编码、状态、日期范围筛选
   - 批量选择 + 批量审批通过/驳回

2. **TransferApply.tsx**（发起调拨申请页）
   - 资产选择器（从已有资产列表选择，支持多选）
   - 目标部门 + 目标员工（级联选择）
   - 调拨原因（文本域）
   - 计划调拨日期（日期选择）

3. **TransferDetail.tsx**（调拨记录详情页）
   - 显示完整调拨信息 + 审批历史

#### API 客户端
- `api/transfer.ts`：`applyTransfer`, `getTransfers`, `getPendingTransfers`, `approveTransfer`, `rejectTransfer`, `getAssetTransferHistory`

#### 菜单
在现有菜单体系中，新增「资产调拨」入口（与「资产领用」「资产借用」并列）

## 5. 验收标准

| ID | 验收条件 | 验证方式 |
|----|---------|---------|
| AC1 | 员工可选择多个 `IN_USE` 状态的资产，发起调拨申请，填写目标部门/员工和原因 | UI 操作 + API POST `/api/transfers/apply` 返回 201 |
| AC2 | 审批人可在「待我审批」列表看到调拨申请，可审批通过或驳回（需填写备注） | UI 操作 + API 审批后 TransferRecord 状态变更 |
| AC3 | 审批通过后，资产的 `assignee` 自动更新为目标员工 | 查询 `Asset.assignee` 确认 |
| AC4 | 同一资产有待处理调拨申请时，不能再次发起新的调拨申请 | API 拒绝重复申请，返回 400 |
| AC5 | 可查询某资产的完整调拨历史记录 | GET `/api/transfers/asset/{assetId}` |
| AC6 | 资产调拨记录永久保留，不可删除 | 数据库 `asset_transfer_record` 表无 delete 操作 |
| AC7 | 借用中（`BorrowRecord` 状态=BORROWED）的资产不能调拨 | API 校验并拒绝 |

## 7. 技术约束

- 审批流程使用现有的 `ApprovalRequest` 框架，`ApprovalType` 新增 `TRANSFER`
- Flyway 已禁用，涉及新表需要手动 DDL 创建
- 资产调拨记录使用软删除模式（`deleted` 字段），物理删除由管理员操作
- 通知类型新增 `TRANSFER_APPROVED`、`TRANSFER_REJECTED`、`TRANSFER_REQUIRED`
