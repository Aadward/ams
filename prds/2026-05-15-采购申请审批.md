# PRD: 资产采购申请审批

## 1. 背景与目标

**问题**: AMS 目前覆盖了资产"入库后"的全生命周期管理（领用/调拨/维修/借用/折旧/盘点/报废），但"采购→入库"这一起点环节缺失。实际业务中，资产先有采购申请 → 审批通过 → 采购入库建档，才能进入后续管理流程。

**目标**: 新增采购申请提交 → 审批 → 通过后自动创建资产档案（待入库状态）的完整闭环，使 AMS 覆盖资产全生命周期。

## 2. 功能范围

### In Scope
- 采购申请表单（申请人、资产名称/类别/数量/预估价格/用途/供应商）
- 采购申请审批流程（Manager/Admin 审批）
- 审批通过后：自动创建资产档案（状态=IN_STOCK，待完善资产信息）
- 采购申请列表（我发起的 / 待我审批 / 全部）
- 审批拒绝：通知申请人

### Out of Scope
- 采购订单PDF生成
- 供应商管理（后续独立模块）
- 实际采购入库流程（仅自动建档，不含付款/物流）
- 多级审批（暂定一级审批）

## 3. 用户故事

| 角色 | 场景 | 预期结果 |
|------|------|----------|
| Employee | 提交采购申请 | 系统记录申请，通知Manager审批 |
| Manager | 收到审批通知，点击审批 | 可以通过或拒绝，拒绝需填理由 |
| Manager | 审批通过 | 系统自动在资产模块创建"待完善"资产记录，申请人收到通知 |
| Admin | 登录系统 | 可以在资产列表看到刚创建的采购资产 |

## 4. 详细需求

### 4.1 后端

**Entity: ProcurementRequest**
```
id: Long (PK)
requestNo: String (编号，格式 PROC-YYYYMMDD-XXXX)
requesterId: Long (FK → Employee)
assetName: String
category: String
quantity: Integer
estimatedPrice: BigDecimal
purpose: String (用途)
supplier: String (供应商)
status: ProcurementStatus (PENDING/APPROVED/REJECTED)
approvalId: Long (FK → ApprovalRequest，可选）
createTime: LocalDateTime
updateTime: LocalDateTime
```

**Enum: ProcurementStatus** (新增)
- PENDING, APPROVED, REJECTED

**ApprovalType 已有**: PROCUREMENT（占位，需激活使用）

**Service: ProcurementService**
- `createProcurementRequest(ProcurementRequestDTO)` → 保存申请，发送 APPROVAL_REQUIRED 通知给 Manager
- `approve(Long procurementId, Long approverId)` → 更新状态=APPROVED，通过 ApprovalService 创建 ApprovalRequest，调用 AssetService 建立资产记录，发送 APPROVAL_APPROVED 通知
- `reject(Long procurementId, Long approverId, String reason)` → 更新状态=REJECTED，发送 APPROVAL_REJECTED 通知
- 分页查询（按状态、申请人筛选）

**Controller: ProcurementController**
- `POST /api/procurements` — 新增采购申请
- `GET /api/procurements` — 分页列表（参数：status, requesterId, page, size）
- `GET /api/procurements/{id}` — 详情
- `POST /api/procurements/{id}/approve` — 审批通过
- `POST /api/procurements/{id}/reject` — 审批拒绝

**与审批流程集成**
- 复用现有 `ApprovalRequest` 实体，通过 `ApprovalType.PROCUREMENT` 标识
- 复用现有通知机制（NotificationType.APPROVAL_REQUIRED / APPROVAL_APPROVED / APPROVAL_REJECTED）

### 4.2 前端

**页面**
- `ProcurementList.tsx` — 采购申请列表（3个Tab：我发起的 / 待我审批 / 全部）
- `ProcurementApply.tsx` — 采购申请表单（提交）
- `ProcurementDetail.tsx` — 采购申请详情 + 审批操作（通过/拒绝按钮，仅审批人可见）

**API**
- `procurementApi.create()` / `procurementApi.list()` / `procurementApi.approve()` / `procurementApi.reject()`

**菜单**
- 左侧菜单新增"采购管理"菜单项（Employee可见申请提交，Manager/Admin额外可见审批Tab）

**状态同步**
- 枚举 ProcurementStatus 需与前端 `procurementApi` 返回的 status 字段对齐

## 5. 验收标准

1. Employee 提交采购申请 → 数据库 procurement_request 表有记录，NotificationType.APPROVAL_REQUIRED 通知已发送
2. Manager 审批通过 → procurement_request.status=APPROVED，asset 表有对应记录（assetName 填充，状态=IN_STOCK）
3. Manager 拒绝 → procurement_request.status=REJECTED，申请人收到 APPROVAL_REJECTED 通知
4. 采购申请列表分页正常，Tab 切换正常
5. `mvn compile -q` 成功，`npm run build` 成功
6. Docker 重建后 API 正常（`curl /api/procurements` 返回空数组或正确数据）

## 6. 优先级

**P0** — 核心流程完整（申请→审批→资产创建）
**P1** — 列表 + 详情页面
**P2** — 通知集成 + 前端状态同步
