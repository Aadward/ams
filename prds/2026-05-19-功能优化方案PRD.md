## 2. P0 级优化

### 2.1 采购审批完整实现

**现状**: PRD 已写好，但后端 Controller/Service 未创建，前端页面未实现。

**实现内容**:

| 组件 | 路径 | 说明 |
|------|------|------|
| Entity | `ProcurementRequest.java` | 采购申请单 |
| Enum | `ProcurementStatus.java` | PENDING / APPROVED / REJECTED |
| Repository | `ProcurementRequestRepository.java` | JPA 查询 |
| Service | `ProcurementService.java` | 申请/审批/查询逻辑 |
| Controller | `ProcurementController.java` | REST API |
| 前端页面 | `ProcurementList.tsx` | 列表（我发起的/待我审批/全部） |
| 前端页面 | `ProcurementApply.tsx` | 申请表单 |
| 前端页面 | `ProcurementDetail.tsx` | 详情+审批操作 |
| 菜单入口 | AppMenu.tsx | 新增"采购管理"菜单 |

**API 设计**:

| Method | Path | 描述 |
|--------|------|------|
| POST | `/api/procurements` | 创建采购申请 |
| GET | `/api/procurements` | 分页列表（status, requesterId） |
| GET | `/api/procurements/{id}` | 详情 |
| POST | `/api/procurements/{id}/approve` | 审批通过 |
| POST | `/api/procurements/{id}/reject` | 审批拒绝 |

**核心逻辑**:
- 申请提交 → 发送 `NotificationType.APPROVAL_REQUIRED` 通知
- 审批通过 → 自动在 `asset` 表创建记录（状态 IN_STOCK，资产名称从申请填充）
- 审批拒绝 → 发送 `NotificationType.APPROVAL_REJECTED` 通知

**验收标准**:
1. `mvn compile` 通过，`npm run build` 通过
2. 申请提交 → `procurement_request` 表有记录
3. 审批通过 → `asset` 表有对应记录
4. Docker 重建后 API 正常

### 2.2 JWT 登录认证真正落地

**现状**: 后端有 `AuthController`，但 userId 在前端硬编码为 1。

**实现内容**:

| 组件 | 说明 |
|------|------|
| `AuthController.login()` | 返回 JWT Token（包含 userId/username/role） |
| `JWT Filter` | 解析 Token，设置 SecurityContext |
| 前端 `AuthProvider` | 登录后保存 token 到 localStorage |
| `api/index.ts` | 封装请求拦截器，自动带上 `Authorization: Bearer <token>` |
| `App.tsx` | 从 localStorage 读取 userId，传递给 NotificationBell |
| 受保护路由 | 未登录时跳转到登录页 |

**API 设计**:

| Method | Path | 描述 |
|--------|------|------|
| POST | `/api/auth/login` | 登录（username + password），返回 JWT |
| GET | `/api/auth/me` | 获取当前用户信息 |

**数据模型**:

```java
// LoginRequest
{ username: String, password: String }

// LoginResponse
{ token: String, userId: Long, username: String, role: String }
```

**验收标准**:
1. 登录页 POST `/api/auth/login` 返回有效 JWT
2. 登录后所有 API 请求自动带上 Token
3. Token 过期或未登录 → 401 或重定向登录页
4. NotificationBell 使用真实 userId

## 3. P1 级优化

### 3.1 统一审批流程

**现状**: 领用/归还用 `ApprovalService`，借用审批在 `BorrowService` 内部，调拨审批在 `AssetTransferService` 内部。

**目标**: 抽象通用审批引擎，所有审批统一入口。

**设计**:

```
ApprovalWorkflow (通用审批工作流)
├── trigger(type, businessId, applicantId)
│   ├── 创建 ApprovalRequest (type=XXX, businessId, requesterId)
│   └── 发送 APPROVAL_REQUIRED 通知
├── approve(approvalId, approverId)
│   ├── 更新 ApprovalRequest.status = APPROVED
│   ├── 执行业务侧回调 (如 AssetService.assign())
│   └── 发送 APPROVAL_APPROVED 通知
└── reject(approvalId, approverId, reason)
    ├── 更新 ApprovalRequest.status = REJECTED
    └── 发送 APPROVAL_REJECTED 通知

业务侧改造:
- BorrowService: 移除内部审批逻辑，改为调用 ApprovalWorkflow
- AssetTransferService: 同上
- 领用/报废: 已有 ApprovalService，继续沿用或迁移到 Workflow
```

**改造范围**:

| 文件 | 改动 |
|------|------|
| `ApprovalWorkflow.java` (新) | 通用审批引擎 |
| `ApprovalService.java` | 重构为调用 Workflow |
| `BorrowService.java` | 移除内部审批，调用 Workflow |
| `AssetTransferService.java` | 同上 |
| `BorrowController.java` | 移除 approve/reject 端点 |
| `AssetTransferController.java` | 同上 |
| `ApprovalList.tsx` | 统一审批列表 |

**验收标准**:
1. 所有审批（领用/借用/调拨/维修/采购）统一走 `ApprovalList` 页面
2. 审批通过/拒绝后业务状态正确更新
3. 通知正确发送

### 3.2 员工自助首页

**现状**: 仪表盘只有统计卡片，员工每次要找列表页看自己的资产/申请。

**实现内容**:

在 Dashboard 增加"我的"Tab 或专区：

| 区块 | 内容 |
|------|------|
| 我的资产 | 当前用户领用/借用的资产列表（数量+快捷入口） |
| 我的申请 | 我发起的采购/借用/调拨申请（状态+时间） |
| 我的待审批 | 需要我审批的任务（数量+快速处理入口） |
| 最近操作 | 我操作的资产日志（最近5条） |

**API 扩展**:

| Method | Path | 描述 |
|--------|------|------|
| GET | `/api/assets/my` | 我的资产（assigneeId = 当前用户） |
| GET | `/api/approvals/my-requests` | 我发起的审批 |
| GET | `/api/approvals/pending-count` | 待我审批数量 |

**验收标准**:
1. 登录员工在首页能看到自己的资产和申请
2. 点击快捷入口直接跳转到对应列表（已筛选）

## 4. P2 级优化

### 4.1 菜单按业务域重新分组

**现状**: 菜单按流程类型组织，借用/调拨/维修/审批混在一起。

**优化后分组**:

```
资产域
├── 资产管理（列表/新建）
├── 借用管理（记录/申请）
├── 调拨管理（记录/申请）
└── 报废管理

运维域
├── 维修记录
├── 定期盘点（计划/任务/报告）
└── 维保管理

供应链域
├── 采购管理
├── 供应商管理
└── 低值易耗品

财务域
├── 折旧台账
├── 保险管理
└── 备份管理

系统域
├── 部门管理
├── 员工管理
├── 审批请求
├── 通知中心
└── 统计报表
```

**图标补全**: 部门管理→`TeamOutlined`，保险管理→`SafetyOutlined`，折旧台账→`AccountBookOutlined`

---

### 4.2 借用 vs 调拨区分优化

**现状**: 两者功能相似，员工容易混淆。

**优化方向**:
- 调拨申请表单增加"是否为临时调拨"选项
- 临时调拨有计划归还日期，期末自动提醒
- 永久调拨直接变更归属，不可逆

或者：
- 将两者合并为"资产变动申请"，子类型区分（借用/调拨）
- 统一列表，筛选器区分类型

**推荐方案**: 第二个（合并），减少重复页面。
